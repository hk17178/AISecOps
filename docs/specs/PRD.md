## 1. 项目概述

### 1.1 一句话
> 给**我们自己小公司 / 部门**（10-100 人规模）搭一个 **AI 驱动的内部安全运营工具**，把 SOC 重复劳动自动化、把分析师从告警海里捞出来。**自建自用，不卖**。

### 1.2 三件事我们想达到

1. **降低我们自己的 SOC 痛点**：告警分诊不用人肉刷、事件调查不用满世界查日志、报告不用复制粘贴
2. **学透 AISecOps 架构**：从立项到上线一遍走完，未来这个领域有什么动静都能看懂
3. **保留可演进性**：哪天要扩展（多团队 / 商业化）能改而不必推翻

### 1.3 不是为了

- ❌ 卖给客户（短期内）
- ❌ 拿合规认证（ISO 27001 / SOC2 / 等保）
- ❌ 对标 Charlotte / Copilot 在功能上 1:1 跟进
- ❌ 7×24 SLA

## 2. 使用者与角色

### 2.1 规模假设
- **10-100 人**的小公司 / 部门
- 安全相关同事 **1-5 人**
- 主要用户：**你自己 + 几个同事**

### 2.2 三个内部角色（够用就行）

| 角色 | 谁 | 主要做什么 |
|---|---|---|
| **管理员** | 你 + 1-2 个核心同事 | 配置平台、配 Agent / Prompt / 工单流程、HITL 审批高风险动作 |
| **分析师** | 安全相关同事 | 看告警、做事件调查、Chat 提问、批量处置 |
| **普通查看** | 其他同事（产品 / 运营 / 高管） | 看仪表盘、自己关注的告警 |

> 不要权限矩阵搞得复杂，3 个角色 + 简单 RBAC 够用。

## 3. 核心场景（按真实用频排序）

### 3.1 ⭐ 场景 A：告警分诊（最高频）
**痛点**：现在告警刷屏没人看，真威胁淹没在误报里。

**预期**：
- 告警进来 5 分钟内出分诊结果（真威胁 / 误报 / 需要人审）
- 真威胁 → 推送通知 + 提供下一步建议
- 误报 → 静默归档但保留可查
- 需要人审 → 进 HITL 队列

**够用就行**：不追求"业界 +77% 准确率"，**比当前人肉好就 OK**。

### 3.2 ⭐ 场景 B：事件调查（次高频）
**痛点**：出事时分析师需要在 5 个系统里查日志、画时间线，半天才出报告。

**预期**：
- Chat 提问"这次入侵的攻击链是什么？"
- 系统自动查 SIEM / EDR / TI / CMDB，给出时间线 + IoC + 影响主机
- 单次调查 30 分钟 → 5 分钟

### 3.3 场景 C：自动报告
**痛点**：日报 / 周报 / 事件报告手写到吐。

**预期**：
- 按模板自动生成（日报 / 周报 / 事件复盘）
- 受众分级（管理层简版 / 技术层详版）
- 一键导出 Markdown / DOCX

### 3.4 场景 D：自动处置（保守开启）
**痛点**：发现真威胁还要手动封 IP / 隔离 / 通知运维。

**预期**：
- 配置 Playbook（如"EDR 高危告警 + 横移迹象 → 隔离"）
- **必须经 HITL 审批**（自用无法承担误封）
- 处置完成后通知 + 审计留痕

### 3.5 不在 V1.0 范围
- 威胁狩猎（主动找潜伏威胁）→ V2 视实际需求做
- UEBA → V2 视数据量做（自用数据量未必够）
- 合规报表（等保 / 个保法）→ 不做，除非内部强制
- 漏洞管理 → 接入 MCP 就行，不做完整功能

## 4. 功能需求

按 12 层组织。每条标优先级：**M**（必做）/ **S**（应做）/ **C**（可做）/ **W**（不做）。

### 4.1 L01 人机交互
- [M] F-101 告警面板（实时流 + 过滤 + 批量标注）
- [M] F-102 工单 UI + HITL 审批队列
- [M] F-104 Chat 面板（自然语言提问）
- [S] F-103 报表 UI（仪表盘 + 日报 / 周报）
- [C] F-105 移动端只读（看告警 + HITL 审批）

### 4.2 L02 Agent + 平台核心
- [M] F-201 Orchestrator + Triage + Investigation + Responder + Reporter 5 个 Agent
- [M] F-203 SOAR Playbook 引擎（含 HITL）
- [M] F-204 Memory Store（先 Postgres 实现）
- [M] F-205 Prompt 版本化（git 管理）
- [M] F-207 通知（企微 / 钉钉 / 邮件 任选一种起步）
- [S] F-202 工单引擎
- [S] F-206 内部 RBAC（3 角色）
- [C] F-208 Intel Agent / Tuning Agent（看数据量再上）

### 4.3 L03 RAG + L04 模型
- [M] F-301 RAG 流水线（含 Reranker）
- [M] F-401 LLM 接入（本地 OpenAI-compat + SaaS 各 1）
- [S] F-403 Skills 库（SOP 模板）
- [C] F-402 DSLM（**P3+ 视数据量评估**，初期不做）
- [C] F-404 知识图谱（视场景）

### 4.4 L05 Gateway + L06 MCP
- [M] F-501 LLM Gateway（路由 / 计费 / 脱敏 / 出域开关 / stub / 降级）
- [M] F-601 MCP：Syslog + 你们现用的 SIEM/EDR 二选一 + 通知（共 ≥ 3 个）
- [S] F-602 凭证走 Vault

### 4.5 L07 业务能力（**只做 A/B/C/D 四类**）
- 详见 §3 场景

### 4.6 L08 分析引擎
- [M] F-801 异常检测 + 关联分析（基础）
- [S] F-802 IoC 关联 + Kill Chain（如能接到 TI）
- [C] 攻击图 / UEBA / 失陷研判（视场景）

### 4.7 L09 数据中台 + L10 采集
- [M] F-901 多源采集（Syslog + 文件 + Webhook 三种起）
- [M] F-902 ETL（归一化 + 富化）
- [S] F-903 数据湖（按保留周期，30 天起）

### 4.8 L11 + L12
- [M] F-1101 CMDB（先 Excel / 简单数据库即可）
- [M] F-1201 凭证管理（Vault）
- [M] F-1202 可观测（trace + log + metric）
- [S] F-1203 调度（Cron 起步）

## 5. 非功能需求（自用基线）

### 5.1 性能（务实版）
- 告警分诊端到端 **≤ 5 分钟**
- LLM 调用 P95 **≤ 10 秒**
- 数据采集吞吐 **目标 1-5K EPS**（自用实际可能 100-500 EPS）
- UI 响应 ≤ 3 秒

### 5.2 安全（自身受攻击的）
按 [CONSTRAINTS.md](../../CONSTRAINTS.md) 全部执行（自用同等严格）：
- 输入校验 C-19、Prompt Injection C-20、Output schema C-21
- 凭证零硬编码 C-9、模型权重不入库 C-10、原始日志不入库 C-11
- 审计不可篡改 C-23
- 数据进 LLM 前脱敏 C-32

### 5.3 合规（借鉴而非追认证）
- **不追**：ISO 27001 / SOC2 / 等保 2.0 认证
- **借鉴**：等保 2.0 三级控制项思路、OWASP LLM Top 10（特别 LLM01/02/06/08/09）
- 内部如有合规审计要求时再补对照矩阵

### 5.4 可观测性
- 每次 LLM 调用必带 metadata（C-33）
- Agent 决策 / MCP 调用必有 trace_id
- 仪表盘必有：今日告警量 / 真威胁数 / HITL 待审 / Token 消耗

### 5.5 可重现性
- C-30 全部执行（分诊 / 调查 temperature=0 + seed）

### 5.6 故障与降级
- C-34 LLM 降级链全部执行
- HA / 灾备：**单机 + 定期备份**即可（自用容忍中断）

### 5.7 系统兼容性
- 见 [compatibility.md](compatibility.md)
- 自用：选你们环境能跑的（Rocky / Ubuntu 任选一种），单一部署模式

### 5.8 ⭐ 自用特有：成本控制
- **LLM token 每月预算上限**（默认 ¥500，可调）
- 超预算自动告警 + 降级到便宜模型
- Reranker / Embedding 用本地开源模型（不烧 token）

### 5.9 ⭐ 自用特有：可维护性
- 单人 / 小团队长期维护成本最低
- 代码 + 文档对齐（C-28 docstring + C-15 V2 架构为权威）
- 退出策略：`docs/restart-guide.md`（待 P1 写），3 个月后自己能 5 分钟重新上手

## 6. 范围

### 6.1 In Scope（V1.0）
- L07 场景 A/B/C/D 四个（分诊 / 调查 / 报告 / 自动处置）
- L02 5 个 Agent 角色
- L06 ≥ 3 个 MCP（Syslog + SIEM/EDR + 通知）
- 单一本地部署
- 内部 3 角色 RBAC

### 6.2 Out of Scope（V1.0 不做）
- ❌ 商业化 / 多租户 / SaaS / License / MSSP
- ❌ 合规认证（除非内部强制）
- ❌ UEBA / 威胁狩猎 / 漏洞管理（V2 视需求）
- ❌ DSLM 训练（V3+ 视数据量）
- ❌ 国际化 / 海外 LLM 默认接入（保留出域开关但默认关）
- ❌ 移动端原生 App
- ❌ 主备双活 / 灾备复杂方案

## 7. 业界对标（学习参考，不为跟进）

> 保留作为**学习锚点**，三个月后看看业界又走到哪了。**不强求 1:1 跟进**。

| 维度 | Charlotte | MS Copilot | QAX-GPT | **我们的取舍** |
|---|---|---|---|---|
| Agent 角色 | 5+1 | 50+ | 多 | **5 个起步**（够用就行） |
| 训练数据 | Falcon 百万 | 微软安全图谱 | 百亿日志 | **不做 DSLM**，纯 LLM + 工具调用 |
| HITL | ✅ | ✅ | 🟡 | **✅ 严格做**（自用刚需） |
| 可重现 | ⚠️ | ⚠️ | ⚠️ | **✅ C-30 严格**（自用刚需） |
| Reranker | ✅ | ✅ | ⚠️ | **✅** |
| MCP / 连接器 | Falcon 原生 | 300+ | 国产生态 | **3 个起步**（够用） |
| 部署 | SaaS | SaaS only | 本地 | **单一本地** |

- Charlotte 的"Agentic Detection Triage / Response / Workflows"分工
- Microsoft Phishing Triage 的 +77% 准确率方法
- 启明星辰 "1+1+N"（统一 LLM + AIDK + N 产线智能体）
