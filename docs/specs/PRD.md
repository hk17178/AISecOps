## 1. 项目概述

### 1.1 愿景
做一个**国产、可信、可演进**的 AI 驱动安全运营平台，对标 Microsoft Security Copilot / CrowdStrike Charlotte / 奇安信 QAX-GPT，让中国 SOC 分析师把"30 分钟一条告警"压缩到"5 分钟一条事件"。

### 1.2 价值主张
- **AI 不替代分析师，而是把他们从重复劳动中解放**：告警分诊、报告生成、情报富化全自动
- **可追溯**：每条 AI 决策都附引用源 + 双模型 cross-check + HITL，**不出"幻觉杀生产"事故**
- **可控**：数据脱敏 + 出域开关 + 国产 LLM 优先，合规性可对外证明
- **可扩展**：MCP 生态而非硬编码集成，客户已有的 SIEM/EDR/防火墙都能接入

### 1.3 业界定位

| 维度 | 对标 |
|---|---|
| 整体方案 | Microsoft Security Copilot (Multi-tenant SaaS 风格) + 奇安信 QAX-GPT（国产化合规） |
| Agent 架构 | CrowdStrike Charlotte Agentic SOAR (5+1 角色) |
| 集成生态 | Anthropic MCP + Microsoft Copilot 300+ connectors |
| 模型策略 | 启明星辰 1+1+N（统一 LLM + AIDK + N 产线智能体） |

**差异化（凭什么赢）**：
1. **MCP 生态优先**：业界 2025-2026 战场，国产暂无完整 MCP-first 方案
2. **可重现 + 可解释**：温度=0 + seed + 双模型 cross-check + 引用必备，敢卖给金融/政务
3. **DSLM 飞轮**：从客户实际反馈中持续训练领域小模型（业界稀有）

## 2. 目标用户与部署模式

### 2.1 双部署模式（核心范围）

| 模式 | SaaS 版 | 本地部署版 |
|---|---|---|
| **目标客户** | 中大型互联网企业、零售、制造、运营商 | 金融、政务、国企、央企、军工 |
| **客户规模** | 200-5000 人 IT 团队 | 大型集团（万人以上） |
| **核心驱动** | 降低 SOC 人力成本 / 7×24 覆盖 | 合规 / 数据不出域 / 自主可控 |
| **LLM 选型** | 默认豆包 / 通义；可切 Claude | 必须本地化（Qwen / DeepSeek / 智谱） |
| **MCP 生态** | 平台预置 + 客户自接 | 全本地化预置 |
| **商务模式** | 订阅制（按 Agent 数 / Token 量） | License + 实施 + 年维 |
| **运维责任** | 我方托管 | 客户运维 + 我方支持 |

### 2.2 角色画像

| 角色 | 日常关注 | 核心需求 |
|---|---|---|
| **SOC 一线分析师**（L1） | 告警分诊、误报识别 | 减少误报、快速取证、自动报告 |
| **SOC 二线分析师**（L2） | 事件调查、根因 | 攻击链还原、IoC 关联、威胁狩猎 |
| **SOC 经理 / 总监** | 整体态势、KPI | 仪表盘、MTTD/MTTR、合规报表 |
| **平台管理员** | 系统稳定、Prompt 治理 | Agent 配置、模型切换、HITL 审批 |
| **MSSP 工程师** | 多客户管理 | 多租户、客户隔离、批量处置 |
| **合规官** | 审计、出域、隐私 | 审计日志、数据脱敏、合规报表 |

### 2.3 不是目标用户
- 个人开发者 / 小微企业（合规性 + 成本不划算）
- 不需要 SOC 能力的非安全企业（如纯 ToC SaaS）

## 3. 核心场景（用户故事）

按 L07 六大能力组织。每个场景含：用户 / 触发 / 预期 / 验收。

### 3.1 告警分诊（Alert Triage）
> **用户故事**：作为 L1 分析师，我希望系统自动判断告警是真威胁还是误报，并给出置信度与证据链，让我只看真正需要人审的告警。

- **触发**：SIEM/EDR 推送一条告警到 L10 采集
- **流程**：L10 → L09 → L08 (correlation + ueba_analytics) → L02 Triage Agent → 若高风险则 L02 Responder + HITL
- **预期**：5 分钟内出分诊结果，置信度 ≥ 0.8 自动闭环，0.6-0.8 转 HITL，<0.6 abstain
- **验收**：基于 50+ 历史告警 eval，误报率 ≤ 业界 +77% 准确率（参考 Microsoft Phishing Triage）

### 3.2 事件调查（Investigation）
> **用户故事**：作为 L2 分析师，我希望自然语言提问"这次入侵的攻击链是什么"，系统自动还原时间线 + IoC + 影响面。

- **触发**：分析师在 L01 Chat 面板提问
- **流程**：L02 Orchestrator → Investigation + Enrichment + Intel → L06 (查 SIEM/EDR/TI) → L08 (attack_graph + kill_chain)
- **预期**：单告警调查 30-60 min → 2-5 min（业界基线）
- **验收**：自动生成时间线 + 攻击链图 + IoC 表 + 影响主机列表

### 3.3 威胁狩猎（Threat Hunting）
> **用户故事**：作为 L2 分析师，我想基于一个 TTP 假设主动找潜在的失陷主机。

- **触发**：分析师选定 ATT&CK 技术（如 T1059.001 PowerShell）
- **流程**：L08 (ioc_correlation + compromise_judgment) → L02 Intel 富化 → L09 全文检索
- **预期**：返回可疑主机列表 + 证据链
- **验收**：召回率 ≥ 80%，准确率 ≥ 70%

### 3.4 SOAR 处置（Playbook）
> **用户故事**：作为平台管理员，我想配置"若 EDR 告警 + 横移迹象，自动隔离主机但走 HITL 审批"。

- **触发**：触发条件匹配 + L02 Responder 决策
- **流程**：L02 SOAR Playbook → 双模型 cross-check (C-27) → HITL Channel → L06 调用 EDR MCP 隔离
- **预期**：Playbook 可视化编辑、HITL 必备、审计日志不可篡改
- **验收**：可配置 ≥ 10 个常见处置剧本（封 IP / 禁号 / 隔离 / 调取证据）

### 3.5 UEBA（用户实体行为分析）
> **用户故事**：作为合规官，我想看到内鬼嫌疑名单 + 证据链。

- **触发**：L08 ueba_analytics 周期运行（默认每小时）
- **流程**：L09 行为数据 → L08 UEBA 算法（基线 + 异常） → L02 解释 + 富化 → L01 风险面板
- **预期**：低误报（人工标注 vs 自动一致率 ≥ 85%）
- **验收**：含至少 6 类内鬼指标（数据下载量、登录异常、权限滥用、敏感操作、横移、异常时段）

### 3.6 漏洞 / 合规（Vuln & Compliance）
> **用户故事**：作为合规官，我想一键生成等保 2.0 三级测评报告。

- **触发**：合规官选择标准 + 时间段
- **流程**：L11 资产 + L06 漏洞扫描 MCP + L08 知识抽取 → L02 Reporter → L01 报表 UI
- **预期**：覆盖等保 2.0 / 个保法 / 数安法
- **验收**：报告含全部要求项、可一键导出 docx / pdf

## 5. 非功能需求

### 5.1 性能
- 单告警分诊端到端 ≤ 5 分钟（业界基线）
- LLM 调用 P95 ≤ 8 秒
- 数据采集吞吐 ≥ 50K EPS（单节点）
- 平台 UI 响应 ≤ 2 秒

### 5.2 安全（自身受攻击的）
- 所有外部输入 schema 校验（C-19）
- Prompt Injection 防护（C-20）
- 凭证零硬编码（C-9）
- 审计日志不可篡改（C-23）
- 数据进 LLM 前脱敏（C-32）
- 详见后续**威胁建模文档**（P2 待补）

### 5.3 合规（卖给客户的）
- 等保 2.0 三级（必须）
- 个保法 / 数安法 / 网安法（必须）
- ISO 27001（SaaS 模式必须）
- 跨境数据传输（默认拒绝出域，开关由客户控制）

### 5.4 可观测性
- 每次 LLM 调用必带 metadata（C-33）
- 每次 Agent 决策、MCP 调用必有 trace_id
- MTTD / MTTR / 误报率 仪表盘

### 5.5 可扩展性
- MCP 接入新工具：≤ 1 天工作量
- 新增 Agent 角色：≤ 0.5 天
- 多租户隔离（SaaS 必须）
- 水平扩展（数据中台 + Gateway）

### 5.6 可重现性
- 高风险决策必须可重现（C-30）
- 推理任务 temperature=0 + seed 固定

### 5.7 故障与降级
- LLM 降级链：备用 model → 规则引擎 → 人审（C-34）
- 单 LLM 故障不影响平台核心可用性

### 5.8 系统兼容性（硬性承诺）
- **OS**：Rocky/RHEL/AlmaLinux 9.4+ / Ubuntu 22.04 LTS / 24.04 LTS（x86_64）
- **编排**：SaaS 必须 K8s 1.28+；本地模式 K8s 与 docker-compose v2.20+ 二选一
- **离线**：本地模式必须支持完全离线部署
- **完整矩阵与版本要求**：见 [compatibility.md](compatibility.md)（权威）

## 6. 范围

### 6.1 In Scope（V1.0）
- 双部署模式（SaaS + 本地）

- 国际化（先中文 + 英文界面，其他语言后置）
- 私有大模型预训练（先用开源 + fine-tune）

## 9. 假设、约束、依赖

### 9.1 假设
- 国内 LLM（豆包 / 通义 / 智谱）能力持续提升，能满足 SOC 场景
- 客户愿意接受"AI + HITL"而非"AI 全自动"
- MCP 协议在 2026 成为业界事实标准

### 9.2 约束
- 团队规模：初期 1-3 人（用户 + AI 辅助）
- 预算：以 LLM API 费用为大头，需控制
- 时间：V1.0 目标 6 个月（详见开发计划）
- 法规：必须满足国内合规

### 9.3 依赖
- 国产 LLM API 稳定性（豆包等）
- 开源 DSLM 基础模型（Qwen / DeepSeek 等）
- 数据中台底层（Kafka / ClickHouse / MinIO）
- MCP Python SDK（Anthropic）
